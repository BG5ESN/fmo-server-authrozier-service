# FMO V4 Root CRL Format

## 1. 作用范围

本文档定义 FMO V4 中 Root CRL 的独立格式。

Root CRL 仅用于吊销 Intermediate CA 证书，不用于吊销 User Certificate。User Certificate 的吊销应由对应的 Intermediate CRL 处理。

除本文另有说明外，编码、base64url、确定性 CBOR、Ed25519、时间戳与字段规范化规则均继承自主协议 V4.0 Protocol — Signatures & Certificates。

## 2. 设计目标

Root CRL 应满足以下目标：

1. 可由 Root 私钥离线签名。
2. 可由设备或服务端独立下载、缓存和验签。
3. 可稳定指向某一张 Intermediate CA 证书，而不是模糊指向某个名称。
4. 可在 JSON、HTTP 和文件存储中展示，但签名依据始终是固定顺序的 CBOR array。
5. 结构尽量简单，便于嵌入式设备实现。

## 3. 通用规则

1. Root CRL 是公开对象，不属于秘密数据。
2. Root CRL 的签名算法固定为 Ed25519。
3. Root CRL 的公开签名输入固定为 CBOR array，不直接对 JSON 文本签名。
4. Root CRL 中的 `signature` 字段不进入被签名数组。
5. `issuerSn`、`crlNumber`、`thisUpdate`、`nextUpdate`、`sn`、`revokedAt`、`reason` 都使用 CBOR integer。
6. `certFingerprint` 在签名数组中使用 32 字节原始值的 CBOR byte string；JSON 展示层使用 base64url 文本。
7. `entries` 在签名数组中作为嵌套 array 写入，条目顺序固定。
8. 条目应先按 `sn` 升序排序；若 `sn` 相同，再按 `certFingerprint` 的字节序升序排序。
9. Root CRL 必须使用确定性 CBOR 编码；不允许 indefinite length。

## 4. Root CRL 对象

推荐 JSON 展示格式如下：

```json
{
  "type": "rootCRL",
  "issuerSn": 1,
  "crlNumber": 3,
  "thisUpdate": 1783291391,
  "nextUpdate": 1783377791,
  "entries": [
    {
      "sn": 1001,
      "certFingerprint": "bSgD0M6Q4fNQHPO8R6N9G8c9Yj0I-9YQYI7WnXv9n9A",
      "revokedAt": 1783291400,
      "reason": 2
    }
  ],
  "signatureAlgorithm": "Ed25519",
  "signature": "root_crl_signature_base64url"
}
```

字段定义如下：

1. `type`：固定为 `rootCRL`。
2. `issuerSn`：签发该 Root CRL 的 Root 证书序列号。
3. `crlNumber`：单调递增的 CRL 版本号，用于缓存替换和更新判断。
4. `thisUpdate`：本份 CRL 的生成时间。
5. `nextUpdate`：建议下次获取新 CRL 的时间上限。
6. `entries`：吊销条目列表，可为空数组。
7. `signatureAlgorithm`：固定为 `Ed25519`。
8. `signature`：对 Root CRL 签名数组做确定性 CBOR 编码后的 Ed25519 签名，展示为 base64url。

## 5. 条目格式

每个 Root CRL 条目表示“某张 Intermediate CA 证书已被吊销”。

推荐 JSON 展示格式如下：

```json
{
  "sn": 1001,
  "certFingerprint": "bSgD0M6Q4fNQHPO8R6N9G8c9Yj0I-9YQYI7WnXv9n9A",
  "revokedAt": 1783291400,
  "reason": 2
}
```

字段定义如下：

1. `sn`：被吊销的 Intermediate CA 证书序列号。
2. `certFingerprint`：被吊销证书的 SHA-256 指纹，用于把同一 `sn` 精确绑定到某一张具体证书。
3. `revokedAt`：该证书生效吊销的时间戳。
4. `reason`：吊销原因码，使用整数枚举。

## 6. 证书指纹定义

`certFingerprint` 必须按以下规则计算：

1. 先按主协议中 Intermediate CA 证书的签名数组规则，构造该证书的固定顺序语义数组。
2. 对该数组做确定性 CBOR 编码，得到原始字节串。
3. 对该字节串计算 SHA-256。
4. 在签名数组中使用这 32 字节原始摘要；在 JSON 展示层使用不带 padding 的 base64url 文本。

说明：

1. 不允许对证书 JSON 文本直接做 SHA-256。
2. 不允许把 `signature` 字段并入 `certFingerprint` 的计算输入。
3. 这样定义后，不同实现即使 JSON 缩进、空格、字段顺序不同，只要证书语义相同，得到的指纹就完全一致。

## 7. 吊销原因码

`reason` 推荐使用以下整数枚举：

| 数值 | 含义 |
| ---- | ---- |
| 0 | unspecified |
| 1 | keyCompromise |
| 2 | caCompromise |
| 3 | superseded |
| 4 | privilegeWithdrawn |
| 5 | cessationOfOperation |

说明：

1. `keyCompromise` 表示中间 CA 私钥疑似泄露或已确认泄露。
2. `caCompromise` 表示中间 CA 体系或签发环境失陷。
3. `superseded` 表示该中间 CA 已被新的中间 CA 证书替代。
4. `privilegeWithdrawn` 表示 Root 主动撤销该中间 CA 的签发权限。
5. `cessationOfOperation` 表示该中间 CA 停止运营或不再提供服务。

## 8. Root CRL 签名数组

Root CRL 的签名数组定义如下：

```text
["FMO", 4, "rootCRL", issuerSn, crlNumber, thisUpdate, nextUpdate, [[sn, certFingerprintBytes, revokedAt, reasonCode], ...]]
```

说明：

1. `protocol` 固定为 `"FMO"`。
2. `version` 固定为整数 `4`。
3. `messageType` 固定为 `"rootCRL"`。
4. 外层 `entries` 是一个嵌套 array。
5. 每个条目固定为 `[sn, certFingerprintBytes, revokedAt, reasonCode]`。
6. `signature` 字段本身不进入该数组。

## 9. 匹配与验证规则

接收端在验证某张 Intermediate CA 证书是否被 Root CRL 吊销时，应执行以下步骤：

1. 验证 Root CRL 的 Ed25519 签名。
2. 检查 `type` 是否为 `rootCRL`。
3. 检查 `issuerSn` 是否与本地信任的 Root 证书一致。
4. 检查当前时间是否已超过 `nextUpdate`；若超过，则应尽快拉取新 CRL。
5. 按主协议规则重建待验证 Intermediate CA 证书的签名数组，并计算其 `certFingerprint`。
6. 在 Root CRL 条目中同时匹配 `sn` 与 `certFingerprint`。
7. 当且仅当 `sn` 与 `certFingerprint` 均匹配时，认定该 Intermediate CA 证书已被吊销。

说明：

1. 仅 `sn` 匹配而 `certFingerprint` 不匹配时，不应直接视为命中；这通常意味着证书重签发、缓存混乱或实现错误。
2. 如果系统明确要求 `sn` 全局不可复用，也仍建议保留 `certFingerprint` 作为二次绑定和审计依据。

## 10. 发布与缓存建议

1. Root CRL 可通过 HTTPS、对象存储或静态文件服务发布。
2. 证书对象中的 `crl` 字段应指向该 Root CRL 的可下载地址。
3. 设备或服务端应缓存最近一次验签通过且 `crlNumber` 最大的 Root CRL。
4. 新下载的 Root CRL 必须先验签，再比较 `crlNumber`，最后替换缓存。
5. 若缓存中的 Root CRL 已过 `nextUpdate` 且无法获取新 CRL，系统可按实现策略降级处理，但应明确记录告警。

## 11. 最小实现建议

如果当前只追求最小可实现版本，至少应做到：

1. 生成和解析 Root CRL JSON。
2. 构造 Root CRL 的 CBOR 签名数组。
3. 使用 Root 私钥离线签名，使用 Root 公钥验签。
4. 对 Intermediate CA 证书计算 `certFingerprint`。
5. 依据 `sn + certFingerprint` 做吊销命中判断。

这样即可在不引入 Delta CRL、OCSP 或在线查询协议的前提下，完成 V4 的 Root 侧中间 CA 吊销闭环。

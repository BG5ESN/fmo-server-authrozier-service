# FMO V4 Protocol — Signatures & Certificates

## 1. 设计目标
V4 的核心原则如下：

1. 证书只负责表达身份与授权边界。
2. 报文只负责表达动作与实时参数。
3. 所有动态动作必须由终端私钥再次签名。
4. 完整 Root 自签名证书通过 OTA 带入设备信任存储，设备运行时以其中的 Root 公钥作为信任锚；Root 不通过 APRS 广播，也不在激活响应中下发。
5. APRS 或其他受限链路上传输 User Certificate 的紧凑表示，完整证书对象可由接收端现场推导。

## 2. 术语

1. Root CA：FMO 根证书，离线保存私钥。
2. Intermediate CA：中间签发者，可按地区或 UID 段授权。
3. User Certificate：用户证书，由中间 CA 签发，只声明“这个呼号、这个 UID、这把公钥”属于被认可用户，并给出有效期。
4. Certificate Chain：Root CA -> Intermediate CA -> User Certificate。
5. CRL：吊销列表。Root CRL 用于吊销中间 CA（使用 SN + 证书指纹进行证书级吊销）；Intermediate CRL 用于吊销用户证书（使用 UID + 证书指纹进行证书级吊销）。

## 3. 通用约定

### 3.1 编码与时间

1. 所有 JSON 使用 UTF-8 编码。
2. 时间戳统一使用 UTC Unix 时间戳，单位为秒。
3. 在 JSON、HTTP 或其他文本链路中，二进制字段统一使用 base64url 编码，不使用普通 base64；若某字段另有传输约定，则以该字段定义为准。
4. 所有大小写敏感字段必须精确匹配。
5. 所有 base64url 编码输出一律不带 padding（即 `=` 字符）。解码实现需兼容带 padding 和不带 padding 两种输入。

### 3.2 签名与摘要

1. 证书签名算法固定为 Ed25519。
2. 报文签名算法固定为 Ed25519。
3. V4 的默认签名输入不是 JSON 文本，而是按协议字段顺序构造的签名数组，再对该数组做确定性 CBOR 编码后得到的字节串。
4. 公开可验证的签名对象必须使用 CBOR array，而不是 CBOR map。字段顺序固定，由协议定义，不允许依赖 JSON 字段顺序或 map key 顺序推导。
5. 所有公开签名对象必须使用确定性 CBOR 编码；不允许使用 indefinite length，也不允许同一语义值存在多种编码形式。
6. JSON、APRS 紧凑文本、HTTP 展示文本都只是传输或展示表示；只要存在 Ed25519 签名，对应签名一律基于同一份 CBOR 签名数组计算，而不是直接对 JSON 或 payload 文本签名。
7. 若某对象定义中明确将 `authTag` 字段标注为通信校验值，则该字段按对象定义处理，不适用 Ed25519/CBOR 签名规则。当前仅 5.5 激活响应中的 `authTag` 属于此类字段，它不是公开可验证签名。
8. 签名数组生成规则如下：
  - 数组元素顺序固定，由协议定义。
  - protocol、messageType 等语义文本使用 CBOR text string。
  - sn、uid、iat、exp、timestamp 等整数使用 CBOR integer。
  - publicKey、nonce、firmwareHash、MAC 等二进制值使用 CBOR byte string。
  - `signature` 字段本身永远不进入被签名数组。
  - 传输层若使用 base64url、ASCII HEX、APRS 紧凑整数等文本表示，验签前必须先还原为相同的语义值，再进入 CBOR 编码。
9. 为降低嵌入式终端的实现复杂度，V4 不要求设备侧实现 JSON 规范化签名；设备侧只需要能按固定顺序组装语义值并做确定性 CBOR 编码即可。
10. 若某些非受限链路实现确实需要对 JSON 对象签名，也必须先将该对象投影为本协议定义的同一份 CBOR 签名数组后再签名，而不是直接对规范化 JSON 字符串签名。

### 3.2.1 字段规范化

进入签名数组之前，各字段应先完成最小必要规范化：

1. callsign 使用大写 ASCII，并作为 CBOR text string 进入签名数组。
2. ssid 先规范化为非负整数，再作为 CBOR integer 进入签名数组。
3. countryCode 使用两位大写 ASCII，并作为 CBOR text string 进入签名数组。
4. MAC 在签名数组中统一表示为 6 字节原始值；若输入来自 `11:22:33:44:55:66` 或 `112233445566` 这类文本，必须先解析为相同的 6 字节 byte string。
5. 所有整数默认使用其语义整数值进入签名数组，不再先转十进制 ASCII。
6. APRS payload 中若为了紧凑传输而把整数编码为 base64、base36 或其他文本槽位，验签前必须先解码回相同整数，再进入签名数组。
7. publicKey、nonce、firmwareHash 等二进制字段在签名数组中统一使用原始字节；其中 firmwareHash 在 JSON/数据库比较时仍可使用 ASCII HEX 文本表示，但进入签名数组前必须先还原为原始 32 字节。
8. `signature` 固定为 64 字节 Ed25519 签名原始值；传输或展示时推荐使用不带 padding 的 base64url 文本。
9. 时间戳统一使用 UTC Unix 时间戳秒值，并作为 CBOR integer 进入签名数组。
10. 自由文本字段在签名数组中直接作为 UTF-8 text string 处理，不再为了分隔符安全而额外转成 base64url。
11. 除非对象定义中明确允许空字段，否则签名数组中不允许出现空语义值。

### 3.2.2 域分离

为避免不同对象之间误验签，所有公开签名结构必须显式包含以下前缀字段：

1. protocol = "FMO"
2. version = 4
3. messageType = 当前对象类型，例如 `userCert`、`rootCA`、`intermediateCA`、`activateReq`、`serverAuthorizerReq`

接收端必须在完全一致的 protocol、version、messageType 条件下重建签名数组，并对其做确定性 CBOR 编码后再验签。

### 3.2.3 CBOR 签名结构示例

签名结构的本质是“固定顺序的语义值数组 + 确定性 CBOR 编码”。

例如：

```text
["FMO", 4, "activateReq", h'112233445566', 1783291391, h'03d9f289c9dad3', "4.0.0", h'5f1c...32bytes', "CN", h'9a81...32bytes']
```

说明：

1. 上述写法是 CBOR diagnostic notation，只用于说明数组结构；实际签名输入是该数组经过确定性 CBOR 编码后的原始字节串。
2. 不存在 JSON 空格、字段排序、缩进、换行差异问题。
3. 实现时应先逐字段生成语义值数组，再一次性做确定性 CBOR 编码。

### 3.3 序列号与引用

1. Root CA 和 Intermediate CA 必须分配全局唯一的 sn。
2. User Certificate 在协议逻辑上可以不携带独立 sn，默认以 pubKeyHash 作为主要引用；若实现侧需要，也可额外分配本地 certSn 作为缓存索引。
3. 查询流程优先携带 pubKeyHash、issuerSn 或链路引用，不携带完整证书。
4. 完整 Root 自签名证书由 OTA 写入设备本地信任存储；设备运行时信任其中的 Root 公钥，不依赖 APRS 或激活响应下发。

### 3.4 私钥边界

V4 的安全边界中，真正需要保密的核心资产只有用户私钥。

1. User Certificate 可以被公开缓存、转发和再分发。
2. APRS 上的紧凑表示本质上也是公开身份证明，不属于秘密数据。
3. 呼号、UID、公钥、issuerSn、有效期都可以被公开看到。
4. 一切信任最终都建立在“中间 CA 对身份字段的签名”和“用户私钥对业务内容的签名”之上。

### 3.5 APRS 证书传输规则

长字段 JSON 只用于签发、存储、调试和非受限链路。

在 APRS-IS 上，V4 不直接传输完整 User Certificate JSON，而是传输一个面向 APRS 的 `CERT blob`。该 blob 是 User Certificate 语义字段的紧凑 CBOR 表示，接收端可据此恢复逻辑上的用户证书身份并验证其合法性。

APRS 证书传输规则如下：

1. APRS payload 使用 `CERT:<base64url blob>` 携带证书紧凑表示。
2. CERT blob 的原始字节必须是确定性 CBOR array，字段顺序固定为 `["FMO", 4, "userCert", issuerSn, callsign, uid, publicKey, iat, exp, caSignature]`。
3. `callsign` 包含在 CERT blob 内，作为被 Intermediate CA 认证的主体标识；APRS 包头中的 callsign 仍需与 blob 内 callsign 做交叉校验。
4. APRS 包头中的 SSID 属于 APRS 会话/发射上下文，不属于 User Certificate 身份字段，不进入 `caSignature` 的签名输入。
5. User Certificate、完整证书链、CRL 地址、license 地址不在 APRS 上直接广播。
6. 接收端不得直接信任 APRS 包头身份；必须以 CERT blob 解码结果为主，并将 APRS 包头 callsign 仅作为一致性交叉校验项。
7. APRS 传播侧推荐直接对固定顺序 CBOR 签名数组签名，而不是对 payload 文本整体签名。

## 4. 证书模型

V4 保留三类证书对象：

1. Root CA 证书
2. Intermediate CA 证书
3. User Certificate

APRS 传播时不再引入独立的“传播证书”对象，而是使用 User Certificate 的紧凑表示。

证书类对象虽然在 JSON 中包含嵌套结构，但进入签名数组时，一律按协议规定拍平成固定顺序 CBOR array。

证书类对象补充规则如下：

1. `isCA` 在签名数组中使用 CBOR boolean，而不是 `1` 或 `0` 文本。
2. `pathLen`、`sn`、`issuerSn`、`uid`、`iat`、`exp`、`uidRange.start`、`uidRange.end` 在签名数组中统一使用 CBOR integer。
3. `name`、`email`、`keyId`、`crl`、`license` 这类自由文本字段在签名数组中统一使用 CBOR text string，不再额外转为 base64url 文本。
4. `publicKey` 在签名数组中统一使用原始公钥字节的 CBOR byte string；JSON 展示层仍使用 base64url 文本。
5. `issuingCountries` 在进入签名数组前，必须先全部转为两位大写国家码，再按字典序升序排序。
6. `issuingCountries` 在签名数组中作为一个嵌套 text string array 写入，而不是“数量 + 多个文本 token”的手工展开形式。

### 4 Root CA 证书

```json
{
  "sn": 1,
  "type": "rootCA",
  "issuer": {
    "name": "bg5esn",
    "email": "xifengzui@yeah.net"
  },
  "subject": {
    "name": "bg5esn",
    "publicKey": "root_public_key_base64url"
  },
  "extensions": {
    "isCA": true,
    "pathLen": 1,
    "crl": "https://bg5esn.com/share/ca/root.crl",
    "license": "https://bg5esn.com/share/ca/license.md",
    "keyId": "root-key-id"
  },
  "iat": 1783291391,
  "exp": 1898999829,
  "signatureAlgorithm": "Ed25519",
  "signature": "root_self_signature"
}
```

说明：

1. Root 自签名证书由服务端预置，并通过 OTA 写入设备信任存储；设备运行时信任其中的 Root 公钥。
2. Root 私钥不得在线使用。

Root CA 证书的签名数组推荐表示如下：

```text
["FMO", 4, "rootCA", sn, issuerName, issuerEmail, subjectName, subjectPublicKeyBytes, isCA, pathLen, crl, license, keyId, iat, exp]
```

说明：

1. Root CA 为自签名证书，但签名数组仍按 `issuer` 和 `subject` 逻辑字段分别展开，不做省略。
2. `issuerName`、`issuerEmail`、`subjectName` 都以原始 UTF-8 文本作为 CBOR text string 进入签名数组。
3. `subjectPublicKeyBytes` 为 Root 公钥的原始 32 字节。
4. `isCA` 对 Root CA 固定为 `true`。
5. `pathLen` 对当前 Root CA 固定为整数 `1`。

### 4.2 Intermediate CA 证书

```json
{
  "sn": 1001,
  "type": "intermediateCA",
  "issuer": {
    "sn": 1,
    "name": "bg5esn",
    "publicKey": "root_public_key_base64url"
  },
  "subject": {
    "name": "bd4wpw",
    "email": "bd4wpw@gmail.com",
    "publicKey": "intermediate_public_key_base64url"
  },
  "extensions": {
    "isCA": true,
    "pathLen": 0,
    "keyId": "intermediate-key-id",
    "crl": "https://bd4wpw.xyz/revoke.crl",
    "license": "https://bd4wpw.xyz/license.md",
    "uidRange": {
      "start": 5000,
      "end": 99999
    },
    "issuingCountries": ["CN", "US", "JP"]
  },
  "iat": 1783291391,
  "exp": 1898999829,
  "signatureAlgorithm": "Ed25519",
  "signature": "signed_by_root"
}
```

说明：

1. uidRange 表示该中间 CA 可签发的 UID 范围。
2. issuingCountries 表示该中间 CA 可签发的国家码范围。
3. pathLen 固定为 0，表示不允许继续下发下级 CA。

Intermediate CA 证书的签名数组推荐表示如下：

```text
["FMO", 4, "intermediateCA", sn, issuerSn, issuerName, issuerPublicKeyBytes, subjectName, subjectEmail, subjectPublicKeyBytes, isCA, pathLen, keyId, crl, license, uidRangeStart, uidRangeEnd, [countryCode1, countryCode2, ...], iat, exp]
```

说明：

1. `issuerSn` 在签名数组中使用整数。
2. `issuerPublicKeyBytes` 和 `subjectPublicKeyBytes` 都使用原始公钥字节，而不是其 base64url 展示文本。
3. `subjectName`、`subjectEmail`、`keyId`、`crl`、`license` 都以原始 UTF-8 文本进入签名数组。
4. `isCA` 对 Intermediate CA 固定为 `true`。
5. `pathLen` 对当前 Intermediate CA 固定为整数 `0`。
6. `issuingCountries` 作为一个已排序的嵌套数组整体进入签名数组。
7. 任一实现只要排序结果不同，就会直接导致签名不一致，因此排序规则必须严格一致。

### 4.3 User Certificate

User Certificate 是终端激活后持有的最小身份证书。它只承载呼号、UID、公钥以及有效期，是后续 APRS 传播和用户自签业务报文的根绑定材料。

```json
{
  "type": "userCert",
  "issuerSn": 1001,
  "subject": {
    "callsign": "BG6VMZ",
    "uid": 12345,
    "publicKey": "user_public_key_base64url"
  },
  "iat": 1783291391, 
  "exp": 1898999829,
  "signatureAlgorithm": "Ed25519",
  "signature": "signed_by_intermediate"
}
```

说明：

1. 中间 CA 只证明“这个呼号、这个 UID、这把公钥”属于被认可用户。
2. 证书本身不承载服务器信息，也不约束用户后续要做什么业务行为。
3. 后续任何业务报文均由 subject.publicKey 对应的私钥自行签名。
4. 若实现侧需要缓存索引，可在本地额外生成 certSn 或 pubKeyHash，但它不是 User Certificate 逻辑字段的必要组成部分。

User Certificate 的签名数组推荐表示如下：

```text
["FMO", 4, "userCert", issuerSn, callsign, uid, subjectPublicKeyBytes, iat, exp]
```

说明：

1. `issuerSn`、`uid`、`iat`、`exp` 在签名数组中使用整数。
2. `callsign` 必须先转为大写 ASCII，再作为 text string 进入签名数组。
3. `subjectPublicKeyBytes` 为用户公钥的原始 32 字节。
4. User Certificate 的 `signature` 由 Intermediate CA 私钥对上述数组的确定性 CBOR 编码结果签名得到。

**User Certificate 指纹（`userCertFingerprintBytes`）**：

`userCertFingerprintBytes` 定义为对 User Certificate TBS 签名数组做确定性 CBOR 编码后的 SHA-256 摘要（32 字节原始值）：

```
userCertFingerprintBytes = SHA-256(deterministicCborEncode([
    "FMO", 4, "userCert",
    issuerSn, callsign, uid,
    subjectPublicKeyBytes,
    iat, exp
]))
```

说明：
1. 指纹计算基于 CBOR 编码结果的字节串，不是对 JSON 文本或 base64url 文本的哈希。
2. `callsign` 进入签名数组前必须先转为大写 ASCII。
3. `subjectPublicKeyBytes` 使用原始 32 字节公钥。

### 4.4 User Certificate 的 APRS 表示

APRS 上广播的并不是另一张独立证书，而是 User Certificate 的 `CERT blob` 紧凑传输表示。其核心含义是：

1. 该 APRS 广播主体对应一个被某个 Intermediate CA 认可过的用户身份。
2. 该用户对应的传播凭证当前仍在有效期内。

CERT blob 解码后的逻辑结构如下：

```json
{
  "type": "userCert",
  "issuerSn": 1001,
  "callsign": "BG6VMZ",
  "uid": 12345,
  "publicKey": "user_public_key_base64url",
  "iat": 1783291391,
  "exp": 1898999829,
  "caSignature": "signed_by_intermediate"
}
```

接收端拿到 CERT blob 后，结合本地缓存的 Intermediate CA 证书，即可验证 `caSignature` 并恢复该广播主体的证书身份。

说明：

1. `callsign` 来自 CERT blob，本身属于被认证内容；APRS 报文头中的 callsign 仅用于交叉校验，不参与定义证书主体。
2. `uid`、`publicKey`、`issuerSn`、`iat`、`exp` 和 `caSignature` 都由 CERT blob 提供。
3. `caSignature` 由 Intermediate CA 对 `callsign`、`issuerSn`、`uid`、`publicKey`、`iat`、`exp` 的 CBOR 签名数组签名得到，不包含 SSID。
4. `exp` 表示有效期，过期则不再信任这个广播身份。
5. 该 APRS 表示只负责让接收端恢复并验证 User Certificate 身份，不负责携带服务器或业务字段。

`caSignature` 的签名数组推荐字段顺序如下：

```text
["FMO", 4, "userCert", issuerSn, callsign, uid, publicKeyBytes, iat, exp]
```

说明：

1. `caSignature` 的签名 TBS 与 User Certificate 的 `toTbsCbor()` 共用同一结构（messageType = `"userCert"`），不引入独立的 `aprsUserCert`。
2. `issuerSn` 紧接协议头之后，`callsign` 次之。
3. `callsign` 必须取自 CERT blob 中的证书主体字段，并先规范化为大写 ASCII。
4. `issuerSn`、`uid`、`iat`、`exp` 在签名数组中都使用整数。
5. `publicKey` 在签名数组中使用原始公钥字节，而不是其 base64url 展示文本。
6. 接收端验签时，必须按完全相同顺序重建数组，并对其做确定性 CBOR 编码后再验签。

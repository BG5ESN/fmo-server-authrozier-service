# FMO V4.0 SAS HTTP 认证（MQTT 内部认证器模式）

## 1. 概述

本文档定义 FMO V4 SAS 的 MQTT 内部认证器模式（以下简称 SAS HTTP 认证器）。

SAS HTTP 认证器作为 EMQX 的内置 HTTP 认证回调运行——由 Broker 在客户端 MQTT CONNECT 时调用 SAS 完成身份验证。

设计背景：

1. 设备直接 MQTT CONNECT，由 EMQX 代理认证调用，无需独立的认证服务通道。
2. EMQX 已具备完整的会话管理能力，SAS 专注于证书链验证和 ACL 生成。

### 1.1 传输协议

SAS HTTP 认证器使用 HTTP/1.1 POST 协议对外提供服务，端点为 `/auth`。选择此协议的原因：

1. EMQX 原生支持 HTTP 密码认证（详见 [EMQX HTTP Auth 文档](https://docs.emqx.com/zh/emqx/latest/access-control/authn/http.html)）。
2. 响应格式使用 JSON，与 EMQX 认证响应规范直接兼容。

---

## 2. 目标与边界

### 2.1 目标

1. 通过完整证书链确认客户端是否为可信 FMO 设备。
2. 通过 Root CRL 和 Intermediate CRL 确认中间证书和用户证书未被吊销。
3. 向通过验证的客户端返回 EMQX 认证允许响应，含 ACL 权限列表。
4. 响应格式支持模板化，便于将来适配 Mosquitto 等其他 Broker。

### 2.2 非目标

1. 不替代 EMQX ACL 引擎。SAS 生成 ACL 列表，EMQX 负责最终鉴权。

---

## 3. 模式选择

所有 V4 服务器均使用 SAS HTTP 认证器模式，设备直接 MQTT CONNECT，username/password 携带证书信息。

设备侧逻辑：

1. 从 APRS STATION 广播解析 `url`、`port`、`uid`、`issuerSn`、`callsign` 以及 STATION 广播者的证书指纹（serverFingerprint）。
2. 设备构造 username=`<callsign>`（大写），password=base64url 编码的认证载荷 JSON，直接发起 MQTT CONNECT 到 `url:port`。

---

## 4. 部署模型

SAS HTTP 认证器模式以单实例运行，仅监听 HTTP 端口供 EMQX 回调：

```
FMO Device
  → APRS STATION 广播发现服务器
  → MQTT CONNECT (username, password=证书+签名) 到 Broker

Server Host
  ├─ MQTT Broker（EMQX，端口 1883）
  │    └─ 配置 HTTP 认证回调: POST http://SAS:8080/auth
  └─ SAS（Docker，HTTP 8080）
       └─ HTTP 监听器 → 证书验证 + ACL 生成
```

建议：

1. SAS HTTP 端点建议监听内网地址（如 `127.0.0.1:8080`），Docker 部署时可映射为 `0.0.0.0`。
2. EMQX 与 SAS 部署在同一主机或同一信任域内。

---

## 5. 协议流程

SAS HTTP 认证器不直接与设备通信，而是由 EMQX 在客户端 CONNECT 时发起 HTTP 调用：

```
设备                EMQX Broker          SAS HTTP Endpoint
 │                      │                      │
 │── MQTT CONNECT ────▶│                      │
 │   username=<CALLSIGN>  │                      │
 │   password=<base64json>│                    │
 │                      │── POST /auth ──────▶│
 │                      │   body: {            │
 │                      │     username,         │
 │                      │     password          │
 │                      │   }                   │
 │                      │                      │
 │                      │           SAS 解析 password JSON
 │                      │           证书链验证 + CRL 检查
 │                      │           timestamp 签名验证
 │                      │           生成 ACL 列表
 │                      │                      │
 │                      │◀─ 200 OK ──────────│
 │                      │   {                  │
 │                      │     result: "allow",  │
 │                      │     is_superuser,     │
 │                      │     client_attrs,     │
 │                      │     expire_at,        │
 │                      │     acl: [...]        │
 │                      │   }                   │
 │                      │                      │
 │◀── CONNACK ────────│                      │
 │                      │                      │
```

## 6. MQTT CONNECT 参数

### 6.1 username

格式：`<CALLSIGN>`（大写 ASCII）

- 为设备呼号。
- 示例：`BG6VMZ`

SAS 收到请求后以 certPackage 中的证书信息为权威来源进行身份校验，username 仅作为辅助标识。

### 6.2 password

password 为一段 base64url 编码的 JSON 字符串（不带 padding），结构如下：

```json
{
  "certPackage": {
    "intermediateCert": {
      "sn": 1001,
      "type": "intermediateCA",
      "issuer": {
        "sn": 1,
        "name": "bg5esn",
        "publicKey": "root_public_key_base64url"
      },
      "subject": {
        "name": "bd4wpw",
        "email": "bd4wpw@gmail.com",
        "publicKey": "intermediate_public_key_base64url"
      },
      "extensions": {
        "isCA": true,
        "pathLen": 0,
        "keyId": "intermediate-key-id",
        "crl": "https://bd4wpw.xyz/revoke.crl",
        "license": "https://bd4wpw.xyz/license.md",
        "uidRange": { "start": 5000, "end": 99999 },
        "issuingCountries": ["CN", "US", "JP"]
      },
      "iat": 1783291391,
      "exp": 1898999829,
      "signatureAlgorithm": "Ed25519",
      "signature": "signed_by_root"
    },
    "userCert": {
      "type": "userCert",
      "certFingerprint": "base64url_sha256_of_tbs_cbor",
      "issuerSn": 1001,
      "subject": {
        "callsign": "BG6VMZ",
        "uid": 50001,
        "publicKey": "user_public_key_base64url"
      },
      "iat": 1783291391,
      "exp": 1898999829,
      "signatureAlgorithm": "Ed25519",
      "signature": "signed_by_intermediate"
    }
  },
  "role": "super",
  "targetCallsign": "BG5ESN",
  "targetUID": 5001,
  "targetUrl": "fmo.bg5esn.com",
  "targetPort": 1883,
  "serverFingerprint": "base64url_sha256",
  "timestamp": 1783291399,
  "proof": {
    "signature": "base64url_ed25519"
  }
}
```

字段说明：

| 字段 | 来源 | 说明 |
|------|------|------|
| `certPackage.intermediateCert` | 设备激活响应 | Intermediate CA 证书 JSON |
| `certPackage.userCert` | 设备激活响应 | User 证书 JSON，certFingerprint 可选 |
| `role` | 设备本地判定 | 设备根据证书指纹匹配判定的角色：`"super"`（与服务器 certFingerprint 一致）、`"admin"`（同一服务器拥有者）、`"user"`（普通用户）。SAS 比对签名数组中的 role 与设备实际角色，不匹配则拒绝 |
| `targetCallsign` | APRS STATION 广播 | 目标服务器呼号，从广播 CERT blob 解析 |
| `targetUID` | APRS STATION 广播 | 目标服务器 UID，从广播 CERT blob 解析 |
| `targetUrl` | APRS STATION 广播 `url` | 目标 MQTT Broker 主机名 |
| `targetPort` | APRS STATION 广播 `P<port>` | 目标 MQTT Broker 端口 |
| `serverFingerprint` | APRS STATION 广播 | 目标服务器用户证书 SHA-256 指纹（32 字节 base64url），从 CERT blob 计算 |
| `timestamp` | 设备本地 | UTC Unix 秒，用于防重放 |
| `proof.signature` | 设备私钥签名 | Ed25519 签名，作用于 §11 定义的签名数组 |

## 7. HTTP 端点规范

### 7.1 请求

**EMQX 配置**：

```
method = post
url = "http://<sas-host>:18084/auth"
headers = { "content-type": "application/json" }
body = { "username": "${username}", "password": "${password}" }
```

EMQX 在客户端 CONNECT 时，将上述 body 以 `POST application/json` 形式发送到 SAS HTTP 端点。

### 7.2 响应（EMQX 模板）

SAS 返回 HTTP 200，Content-Type `application/json`：

```json
{
  "result": "allow",
  "is_superuser": false,
  "client_attrs": {
    "callsign": "BG6VMZ",
    "uid": "50001"
  },
  "expire_at": 1783305800,
  "acl": [
    { "permission": "allow", "action": "all", "topic": "FMO/RAW" },
    { "permission": "allow", "action": "all", "topic": "FMO/TELE" },
    { "permission": "allow", "action": "all", "topic": "FMO/USER_RIG" },
    { "permission": "allow", "action": "publish", "topic": "FMO/QSO/+" },
    { "permission": "allow", "action": "subscribe", "topic": "FMO/QSO/UID/50001" },
    { "permission": "allow", "action": "publish", "topic": "FMO/LATE/UID_V1/50001" },
    { "permission": "allow", "action": "subscribe", "topic": "FMO/LATE/UID_V1/50001" }
  ]
}
```

字段说明（遵循 EMQX HTTP Auth 规范）：

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| `result` | string | 是 | `"allow"` 通过；`"deny"` 拒绝；`"ignore"` 跳过 |
| `is_superuser` | bool | 否 | 超级用户权限，默认 false |
| `client_attrs` | object | 否 | 客户端属性（EMQX v5.7.0+） |
| `expire_at` | int64 | 否 | Unix 秒时间戳，到期后强制断开重新认证（EMQX v5.8.0+） |
| `acl` | array | 否 | 客户端 ACL 权限规则（EMQX v5.8.0+） |

### 7.3 拒绝响应

```json
{
  "result": "deny"
}
```

详细错误信息通过 SAS 日志输出，不暴露给客户端。

## 8. 响应模板

SAS 的 HTTP 响应格式通过模板系统生成，支持不同 Broker 的输出格式。

### 8.1 当前支持

| 模板名 | 适用 Broker | 说明 |
|--------|-------------|------|
| `emqx` | EMQX 5.x | 默认模板，遵循 EMQX HTTP Auth JSON 响应规范 |

### 8.2 配置

```json
{
  "http": {
    "enabled": true,
    "port": 18084,
    "responseTemplate": "emqx"
  }
}
```

### 8.3 扩展

将来若需支持 Mosquitto 等其他 Broker，只需：

1. 实现新的模板类（如 `MosquittoResponseTemplate`）。
2. 在配置中将 `responseTemplate` 改为对应名称。
3. 模板负责将 SAS 内部验证结果（result、role、ACL 列表）映射为目标 Broker 的响应 JSON 格式。

---

## 9. 证书链验证

SAS HTTP 认证器使用证书验证流水线完成设备身份校验。

验证顺序：

1. 反序列化 `certPackage` 中的 Intermediate CA 和 User Certificate。
2. 遍历 Root CA 集，找到能签发 Intermediate CA 的 Root。
3. 按 [Signatures §4.1-§4.3](V4.0%20Protocol%20-%20Signatures%20%26%20Certificates.md) 验证 Root → Intermediate → User 链。
4. 检查 `allowIssuerSn` 白名单策略。
5. 执行两级 CRL 吊销检查（详见 §10）。
6. 验证私钥持有证明（详见 §11）。

---

## 10. CRL 吊销检查

使用本地缓存 + 主动出站 HTTPS 获取。CRL 策略为 fail-soft：

- 若 CRL 未过期 → 正常检查撤销条目。
- 若 CRL 过期且刷新失败 → 告警日志，**继续放行**请求。
- 若 CRL 过期但刷新成功 → 用新 CRL 替换缓存并正常检查。

---

## 11. 私钥持有证明

SAS HTTP 认证器不使用 challenge-response，改用 **timestamp 自签名**。

### 11.1 签名数组（12 元 CBOR）

```
["FMO", 4, "serverAuthorizerReqHttp",
 serverUid, targetCallsign, targetUID,
 role,
 targetUrl, targetPort,
 serverFingerprintBytes,
 timestamp,
 userCertFingerprintBytes]
```

字段说明：

| 位置 | 字段 | CBOR 类型 | 来源 |
|------|------|-----------|------|
| 0 | protocol | text string `"FMO"` | 常量 |
| 1 | version | integer `4` | 常量 |
| 2 | messageType | text string `"serverAuthorizerReqHttp"` | 常量 |
| 3 | serverUid | integer | 服务端配置，SAS 内部已知 |
| 4 | targetCallsign | text string | APRS STATION 广播 CERT blob |
| 5 | targetUID | integer | APRS STATION 广播 CERT blob |
| 6 | role | text string | 设备本地判定：`"super"` / `"admin"` / `"user"` |
| 7 | targetUrl | text string | APRS STATION 广播 `url` 字段 |
| 8 | targetPort | integer | APRS STATION 广播 `P<port>` 字段 |
| 9 | serverFingerprintBytes | byte string (32B) | APRS STATION 广播 CERT blob 指纹 |
| 10 | timestamp | integer | 设备 UTC Unix 秒 |
| 11 | userCertFingerprintBytes | byte string (32B) | User Certificate TBS CBOR SHA-256 |

### 11.2 验证步骤

1. SAS 根据 User Certificate 中的 `publicKey` 验证 `proof.signature`。
2. SAS 验证 `|serverTime - timestamp| ≤ 120` 秒。
3. SAS 判定 User Certificate 对应设备的实际角色（参见 §12.1）。
4. SAS 比对签名数组中的 `role` 与设备实际角色，不匹配则拒绝。设备借此发现自身在服务器上的角色。
5. SAS 验证 `targetCallsign` 等于本地配置的 `server.callsign`。
6. SAS 验证 `targetUID` 等于本地配置的 `server.uid`。
7. SAS 验证 `targetUrl` 等于本地配置的 `mqtt.host`。
8. SAS 验证 `targetPort` 等于本地配置的 `mqtt.port`。
9. SAS 验证 `serverFingerprintBytes` 等于本地服务器用户证书的 SHA-256 指纹。

### 11.3 安全说明

- timestamp 自签名方式攻击窗口 ≤ 120 秒。对业余无线电局域网场景可接受。
- `targetUID`、`targetCallsign` 和 `serverFingerprintBytes` 防跨服务器重放。
- `targetUrl` 和 `targetPort` 防转发攻击——攻击者截获合法 MQTT CONNECT 请求后无法将其转发到其他 EMQX 实例冒领鉴权。
- `userCertFingerprintBytes` 防跨设备冒用。

## 12. ACL 生成

SAS 直接生成 ACL 列表写入 EMQX 响应。

### 12.1 角色判定

设备侧通过证书指纹与服务器归属关系判定角色，发送至 SAS 服务端交叉检验：

| 设备本地判定 | 发送 role | 条件 |
|---|---|---|
| super | `"super"` | 设备 UserCert 指纹 == 服务端 `admins[].certFingerprint`（指纹匹配） |
| admin | `"admin"` | 设备 UID == 服务器 UID（同一服务器拥有者），但指纹不匹配 |
| user | `"user"` | 以上均不满足 |

SAS 服务端验证步骤：
1. 遍历 `server.admins[]` 列表，根据设备用户证书指纹与 UID 确定其服务端角色。
2. 将签名数组中的 `role` 字段与上一步的服务端角色比对。
3. 不匹配时拒绝认证（`role mismatch`），设备据此获知自身在服务器上的真实角色。

角色对应的 ACL 权限：

| 角色 | 黑名单 (MUTELIST) | Profile / Remote Control / Server Info |
|------|-------------------|----------------------------------------|
| super | pub（读写） | pub（读写） |
| admin | pub（读写） | sub（只读） |
| user | sub（只读） | sub（只读） |

### 12.2 输出映射

从 AclStore 解析的 compact 后的 `all`/`pub`/`sub` 三维话题列表映射为 EMQX ACL 规则：

```json
[
  { "permission": "allow", "action": "all", "topic": "FMO/RAW" },
  { "permission": "allow", "action": "publish", "topic": "FMO/QSO/+" },
  { "permission": "allow", "action": "subscribe", "topic": "FMO/QSO/UID/50001" }
]
```

- `all` 话题 → `action: "all"`
- `pub` 话题 → `action: "publish"`
- `sub` 话题 → `action: "subscribe"`

## 13. 错误处理

| 验证失败原因 | HTTP 响应 | SAS 日志 |
|---|---|---|
| 请求格式错误（无法解析） | `result: "deny"` | Warn + 详细错误 |
| 证书链验证失败 | `result: "deny"` | Warn + 具体失败步骤 |
| Intermediate 不在白名单 | `result: "deny"` | Warn |
| 证书已被 CRL 吊销 | `result: "deny"` | Warn |
| timestamp 超时 | `result: "deny"` | Warn |
| 签名验证失败 | `result: "deny"` | Warn |
| 服务器身份不匹配 | `result: "deny"` | Warn |
| 内部异常 | `result: "deny"` | Error + 堆栈 |

SAS HTTP 认证器模式下错误信息不通过响应 body 返回给客户端（避免信息泄露），详细错误仅记录在 SAS 日志中。所有非通过情况统一返回 `result: "deny"`。

## 14. 配置

### 14.1 完整 config.json（HTTP 模式）

```json
{
  "server": { "uid": 5001, "callsign": "BG5ESN", "issuerSn": 1001, "certFingerprint": "..." },
  "mqtt": { "host": "fmo.bg5esn.com", "port": 1883 },
  "trust": { "allowIssuerSn": [], "rootsDir": "~/.sas/roots" },
  "crl": { "refreshSec": 14400 },
  "http": { "addr": "0.0.0.0", "port": 8080, "responseTemplate": "emqx" },
  "log": { "level": "Info" }
}
```

### 14.2 字段说明

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `http.port` | int | `8080` | HTTP 监听端口 |
| `http.ttlSec` | int | `14400` | 会话有效期（秒），到期后 EMQX 强制重新认证 |
| `http.responseTemplate` | string | `"emqx"` | 响应模板名称 |

## 15. 安全考量

1. HTTP 端点**不得直接暴露到公网**，应仅监听 `127.0.0.1` 或内网地址，由 EMQX 内网调用。
2. password 载荷通过 MQTT CONNECT 明文传输时可能被旁路获取；攻击窗口为 timestamp 有效期（≤ 120 秒）加证书链的有效性。建议在受控局域网环境下部署。
3. `proof.signature` 绑定 `targetCallsign`、`targetUID`、`targetUrl`、`targetPort` 和 `serverFingerprintBytes`，防止跨服务器重放与转发攻击。
4. STATION 广播的 serverFingerprint 应与 SAS 本地服务器证书指纹一致，防止广播伪造。

---

## 16. 最小可实现版本

1. 在 SAS 进程中新增 HTTP 监听器（HttpListener），监听 `http.port`。
2. 实现 `POST /auth` 端点，从 request body 接收 EMQX 透传的 `username`（设备 callsign）和 `password`（base64url 认证载荷 JSON，含证书+签名）。
3. 从 `password` JSON 中反序列化 certPackage、targetCallsign、targetUID、targetUrl、targetPort、serverFingerprint、timestamp、proof。
4. 复用证书链验证流水线（CertVerifier + CrlManager）。
5. 按 12 元 CBOR 签名数组验证 Ed25519 proof.signature（含 role 字段）。
6. 验证 timestamp 在 ±120s 窗口内。
7. 验证 targetCallsign / targetUID / targetUrl / targetPort / serverFingerprint 与本地配置一致。
8. 复用 AclStore 生成 ACL 列表。
9. 按 EMQX 模板输出 JSON 响应（result + is_superuser + expire_at + acl）。
10. 非通过情况统一返回 `{"result":"deny"}`。

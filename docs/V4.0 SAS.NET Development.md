## 3. 关键实现要点

### 3.1 CBOR 一致性

**这是整个实现中最容易出问题的点。** .NET `System.Formats.Cbor.CborWriter` 默认行为需要确认为确定性 CBOR（RFC 7049 §3.9），具体要求：

1. 所有整数使用最短编码（`WriteInt64` 会自动选择最短编码）。
2. `WriteStartArray(n)` 中使用 definite-length，不使用 indefinite-length。
3. 排序规则：`issuingCountries` 在签名数组中必须按字典序升序。
4. 推荐在开发初期写一个对比测试：用 C++ 固件侧输出已知 TBS 字节和签名，.NET 侧重建并验签，确认一致。

### 3.2 Ed25519 验签

使用 `Chaos.NaCl.Standard` 的 `Ed25519` 类：

```csharp
using Chaos.NaCl;

bool valid = Ed25519.Verify(signature, message, publicKey);
byte[] sig = Ed25519.Sign(message, privateKey);
```

### 3.3 证书链验证流程

```
1. 解析 certPackage.intermediateCert → IntermediateCaCert
2. 解析 certPackage.userCert → UserCert
3. 遍历 RootCaStore.AllRoots，找到 pubKey 验证 intermediateCert.signature 的 Root
4. root.VerifySelfSignature()
5. intermediate.IsExpired(now) → reject
6. intermediate.VerifyBy(root) → reject
7. user.IsExpired(now) → reject
8. user.Uid ∈ [intermediate.uidRangeStart, intermediate.uidRangeEnd] → reject
9. user.VerifyBy(intermediate) → reject
10. allowIssuerSn 白名单检查 → reject
11. CRL 吊销检查 → reject
12. timestamp 自签名验证 → reject
```

### 3.4 CRL 吊销检查

```
1. RootCRL.VerifyBy(root)            → 用 Root 公钥验 CRL 签名
2. rootCrl.IsRevoked(intermediate.sn, intermediate.fingerprint()) → reject
3. IntermediateCRL.VerifyBy(intermediate) → 用 Intermediate CA 公钥验 CRL 签名
4. intermediateCrl.IsRevoked(user.uid, user.fingerprint()) → reject
```

CRL 从证书 `crl` 字段 URL 拉取，缓存到 `~/.sas/crl/{issuerSn}/`；
`nextUpdate` 过期后刷新失败 → 告警日志，继续放行（fail-soft）。

**非阻塞设计**：CRL 获取不阻塞 HTTP 请求线程。

- `CrlManager.Initialize()` 启动时从本地磁盘加载已缓存的 CRL，网络刷新以 fire-and-forget 方式后台执行。
- `RegisterIntermediate()` 首次遇到新 intermediate 时：内存命中 → 直接返回；本地缓存文件命中 → 同步加载（本地磁盘快速读取）；都未命中 → 触发后台 fetch 任务，不等待，本次请求用已有 CRL 数据做吊销检查。
- 后台定时器（默认 14400s）周期性刷新所有已知 root 和 intermediate 的 CRL。
- 如果某 intermediate 的 CRL 从未获取成功，该 intermediate 签发的用户证书仍可通过认证（无 CRL ≠ 已吊销）。

### 3.5 日志架构

`Logger` 使用 `System.Threading.Channels.Channel<T>` 实现非阻塞异步日志：

- **生产端**：`Logger.Info/Warn/Error/Debug` 调用 `channel.TryWrite()` 入队（纳秒级，永不阻塞调用线程）。
- **消费端**：单个后台长生命周期 `Task` 从 Channel 读取并逐条写入 `Console.Out`（Info/Debug）或 `Console.Error`（Warn/Error），每条写入后显式 `Flush()`。
- **关闭**：`Logger.Flush()` 完成 Channel Writer 并等待消费者排空队列（最多 5 秒），在 `Program.cs` 退出前调用。

此设计确保 HTTP 请求线程在任何日志调用上零等待，即使 stdout 被管道到慢消费者（如 Docker 日志驱动）也不影响请求处理。

### 3.6 HTTP Proof 验证（timestamp 自签名）

签名数组（12 元确定性 CBOR 编码后验签）：

```
["FMO", 4, "serverAuthorizerReqHttp",
 serverUid, targetCallsign, targetUID,
 role,
 targetUrl, targetPort,
 serverFingerprintBytes,
 timestamp,
 userCertFingerprintBytes]
```

验证步骤：Ed25519 验签 → timestamp 在 ±120s 窗口内 → targetCallsign/targetUID/targetUrl/targetPort/serverFingerprint 与本地配置一致 → role 与设备实际角色一致。

详见 [V4.0 SAS HTTP Authentication §11](V4.0%20SAS%20HTTP%20Authentication.md#11-私钥持有证明)。

### 3.7 ACL 生成

ACL 由 `AclStore` 从角色权限文件中加载，运行时按角色名查询并替换 `{uid}` 占位符。角色定义文件位于 `~/.sas/roles/`，首次运行自动从内置目录种子。

**角色文件格式**（JSON，`{uid}` 在运行时替换为实际 UID）：

```json
{
  "role": "super",
  "all": ["FMO/RAW", "FMO/TELE", "FMO/USER_RIG"],
  "pub": ["FMO/QSO/+", "FMO/LATE/UID_V1/{uid}", "FMO/PROFILE", ...],
  "sub": ["FMO/QSO/UID/{uid}", "FMO/LATE/UID_V1/{uid}"]
}
```

**鉴权流程**：`GetRole(userCert)` → `AclStore.GetPermissions(role)` → 替换 `{uid}` → 生成 EMQX ACL 响应

**降级逻辑**：角色文件加载失败时回退到硬编码默认值。普通用户（无角色匹配）使用 `user` 角色。

## 4. 配置

SAS 使用 `~/.sas/config.json` 作为配置文件。

### 启动逻辑

```
SAS 启动
  ├─ ~/.sas/config.json 存在？
  │   └─ 读取文件，启动 HTTP 监听器
  │
  ├─ config.json 不存在 + 无 CLI 参数 + 交互终端？
  │   └─ 进入交互式配置 → 写回 config.json → 启动
  │
  └─ config.json 不存在 + 有 CLI 参数？
      └─ CLI 参数 + 默认值 → 写回 config.json → 启动
```

**交互式配置**：在 Windows 上双击 `sas.exe` 或在终端无参运行，进入引导式问答配置。

### 4.1 config.json

```json
{
  "server": {
    "uid": 0,
    "callsign": "",
    "issuerSn": 0,
    "certFingerprint": "",
    "admins": [{ "uid": 0, "certFingerprint": "", "role": "super" }]
  },
  "mqtt": { "host": "", "port": 1883 },
  "trust": { "allowIssuerSn": [], "rootsDir": "~/.sas/roots" },
  "crl": { "refreshSec": 14400 },
  "http": { "addr": "0.0.0.0", "port": 8080, "ttlSec": 14400, "responseTemplate": "emqx" },
  "log": { "level": "Info" }
}
```

### 4.2 CLI 参数（首次初始化，无 config.json 时生效）

config.json 不存在且有 CLI 参数时，通过参数初始化并写回文件。

| 参数 | 类型 | 映射字段 | 说明 |
|------|------|----------|------|
| `--config <path>` | string | — | 配置文件路径，默认 `~/.sas/config.json` |
| `--server-uid <long>` | long | `server.uid` | **必填**，服务器 UID |
| `--server-callsign <str>` | string | `server.callsign` | **必填**，服务器呼号（大写） |
| `--mqtt-host <str>` | string | `mqtt.host` | **必填**，MQTT Broker 主机名 |
| `--cert-fingerprint <str>` | string | `server.certFingerprint` | **必填**，服务器 UserCert 指纹 base64url |
| `--mqtt-port <int>` | int | `mqtt.port` | MQTT Broker 端口，默认 1883 |
| `--http-port <int>` | int | `http.port` | HTTP 认证回调端口，默认 8080 |
| `--http-addr <str>` | string | `http.addr` | HTTP 绑定地址，默认 0.0.0.0 |
| `--http-ttl <int>` | int | `http.ttlSec` | 会话有效期（秒），默认 14400 |
| `--allow-issuer-sn <str>` | string | `trust.allowIssuerSn` | 逗号分隔，如 `"1001,2001"` |
| `--issuer-sn <long>` | long | `server.issuerSn` | 服务器操作员的 Intermediate CA SN |
| `--roots-dir <path>` | string | `trust.rootsDir` | Root CA 证书目录 |
| `--crl-refresh <int>` | int | `crl.refreshSec` | CRL 刷新间隔，默认 14400 |
| `--log-level <str>` | string | `log.level` | 日志级别：Debug/Info/Warn/Error，默认 Info |

**校验规则**：四必填项（`server.uid`、`server.callsign`、`mqtt.host`、`server.certFingerprint`）在合并 CLI 和 config.json 后仍为空或 0 时拒绝启动。

### 4.3 管理员配置 (server.admins)

SAS 支持多管理员，通过 `server.admins` 数组定义。每个管理员条目需同时匹配 `uid` 和 `certFingerprint`（证书 SHA-256 指纹，base64url 编码）。

**自动填充**：`admins` 为空数组且 `server.certFingerprint` 有值时，自动将初始化用户设为唯一管理员：
```json
"admins": [{ "uid": 2, "certFingerprint": "PHqeCHHC..." }]
```

**降级逻辑**：`admins` 为空且 `server.certFingerprint` 也为空时，降级为仅 `uid == server.uid` 匹配（向后兼容旧配置）。

**多管理员示例**：
```json
"server": {
  "uid": 2,
  "callsign": "BG5ESN",
  "certFingerprint": "PHqeCHHC...",
  "admins": [
    { "uid": 2,    "certFingerprint": "PHqeCHHC...", "role": "super" },
    { "uid": 5001, "certFingerprint": "QWJjRGVG...", "role": "admin" }
  ]
}
```

**角色说明**：

| 角色 | 权限范围 | MUTELIST | PROFILE / REMOTE_CONTROL / SERVER_INFO |
|------|---------|----------|----------------------------------------|
| `super` | 完全管理员（建服者） | pub | pub |
| `admin` | 封号管理员 | pub | sub |
| （无角色） | 普通用户 | sub | sub |

`role` 为空时默认视为 `"super"`，向后兼容。

**ACL 效果**：管理员在 EMQX ACL 响应中获得对应管理话题的 **publish** 权限。普通用户仅能 **subscribe**。

> 管理员配置不支持 CLI 参数，需直接编辑 `config.json`。交互式配置中会给出提示。

## 6. 部署方式

### 6.1 预编译二进制（推荐）

从 Releases 页面下载对应平台的压缩包，解压即用，无需安装 .NET SDK。

### 6.2 自行编译

```bash
# 全部平台打包
./scripts/publish.ps1
# → bin/win-x64.zip, bin/linux-x64.tar.gz, bin/linux-arm64.tar.gz

# 单平台
dotnet publish src/Sas.csproj -r win-x64 --self-contained -p:PublishSingleFile=true -o out
```

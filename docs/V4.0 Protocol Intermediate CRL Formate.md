# FMO V4 Intermediate CRL Format

## 1. 作用范围

本文档定义 FMO V4 中 Intermediate CRL 的独立格式。

Intermediate CRL 仅用于吊销 User Certificate，不用于吊销 Intermediate CA。Intermediate CA 的吊销应由 Root CRL 处理。

除本文另有说明外，编码、base64url、确定性 CBOR、Ed25519、时间戳与字段规范化规则均继承自主协议 V4.0 Protocol — Signatures & Certificates。

## 2. 设计目标

Intermediate CRL 应满足以下目标：

1. 不暴露呼号。
2. 不把 UID 本身永久拉黑，允许同一 UID 后续重新签发新证书。
3. 可由设备或服务端独立下载、缓存和验签。
4. 可稳定指向某一张具体的 User Certificate，而不是模糊指向某个用户文本标识。
5. 结构尽量简单，便于嵌入式设备实现。

## 3. 通用规则

1. Intermediate CRL 是公开对象，不属于秘密数据。
2. Intermediate CRL 的签名算法固定为 Ed25519。
3. Intermediate CRL 的公开签名输入固定为 CBOR array，不直接对 JSON 文本签名。
4. Intermediate CRL 中的 `signature` 字段不进入被签名数组。
5. `issuerSn`、`crlNumber`、`thisUpdate`、`nextUpdate`、`uid`、`revokedAt`、`reason` 都使用 CBOR integer。
6. `certFingerprint` 在签名数组中使用 32 字节原始值的 CBOR byte string；JSON 展示层使用 base64url 文本。
7. `entries` 在签名数组中作为嵌套 array 写入，条目顺序固定。
8. 条目应先按 `uid` 升序排序；若 `uid` 相同，再按 `certFingerprint` 的字节序升序排序。
9. Intermediate CRL 必须使用确定性 CBOR 编码；不允许 indefinite length。

## 4. Intermediate CRL 对象

推荐 JSON 展示格式如下：

```json
{
  "type": "intermediateCRL",
  "issuerSn": 1001,
  "crlNumber": 12,
  "thisUpdate": 1783291391,
  "nextUpdate": 1783377791,
  "entries": [
    {
      "uid": 12345,
      "certFingerprint": "8VxanV1umFl4Jx3y0DjqH2qKX9xZdQ4feJkTz9pO6n0",
      "revokedAt": 1783291400,
      "reason": 1
    }
  ],
  "signatureAlgorithm": "Ed25519",
  "signature": "intermediate_crl_signature_base64url"
}
```

字段定义如下：

1. `type`：固定为 `intermediateCRL`。
2. `issuerSn`：签发该 Intermediate CRL 的 Intermediate CA 证书序列号。
3. `crlNumber`：单调递增的 CRL 版本号，用于缓存替换和更新判断。
4. `thisUpdate`：本份 CRL 的生成时间。
5. `nextUpdate`：建议下次获取新 CRL 的时间上限。
6. `entries`：吊销条目列表，可为空数组。
7. `signatureAlgorithm`：固定为 `Ed25519`。
8. `signature`：对 Intermediate CRL 签名数组做确定性 CBOR 编码后的 Ed25519 签名，展示为 base64url。

## 5. 条目格式

每个 Intermediate CRL 条目表示“某个 UID 名下的一张具体 User Certificate 已被吊销”。

推荐 JSON 展示格式如下：

```json
{
  "uid": 12345,
  "certFingerprint": "8VxanV1umFl4Jx3y0DjqH2qKX9xZdQ4feJkTz9pO6n0",
  "revokedAt": 1783291400,
  "reason": 1
}
```

字段定义如下：

1. `uid`：被吊销证书所属的 UID，用作快速定位键。
2. `certFingerprint`：被吊销 User Certificate 的 SHA-256 指纹，用作精确命中键。
3. `revokedAt`：该证书生效吊销的时间戳。
4. `reason`：吊销原因码，使用整数枚举。

说明：

1. `uid` 不是唯一命中条件；它只是定位和分组字段。
2. 真正的证书级吊销命中条件是 `uid + certFingerprint` 同时匹配。
3. 这样设计后，同一 UID 后续重新签发的新证书不会因为旧 CRL 条目被误伤。

## 6. 用户证书指纹定义

`certFingerprint` 必须按以下规则计算：

1. 先按主协议中 User Certificate 的签名数组规则，构造该证书的固定顺序语义数组。
2. 对该数组做确定性 CBOR 编码，得到原始字节串。
3. 对该字节串计算 SHA-256。
4. 在签名数组中使用这 32 字节原始摘要；在 JSON 展示层使用不带 padding 的 base64url 文本。

说明：

1. 不允许对 User Certificate 的 JSON 文本直接做 SHA-256。
2. 不允许把 `signature` 字段并入 `certFingerprint` 的计算输入。
3. 这样定义后，不同实现即使 JSON 缩进、空格、字段顺序不同，只要证书语义相同，得到的指纹就完全一致。

## 7. 吊销原因码

`reason` 推荐使用以下整数枚举：

| 数值 | 含义 |
| ---- | ---- |
| 0 | unspecified |
| 1 | keyCompromise |
| 2 | superseded |
| 3 | privilegeWithdrawn |
| 4 | cessationOfOperation |

说明：

1. `keyCompromise` 表示用户私钥疑似泄露或已确认泄露。
2. `superseded` 表示旧证书已被新证书替代。
3. `privilegeWithdrawn` 表示 Intermediate CA 主动撤销该用户证书的使用资格。
4. `cessationOfOperation` 表示该证书对应的使用场景已经终止，不再继续信任。

## 8. Intermediate CRL 签名数组

Intermediate CRL 的签名数组定义如下：

```text
["FMO", 4, "intermediateCRL", issuerSn, crlNumber, thisUpdate, nextUpdate, [[uid, certFingerprintBytes, revokedAt, reasonCode], ...]]
```

说明：

1. `protocol` 固定为 `"FMO"`。
2. `version` 固定为整数 `4`。
3. `messageType` 固定为 `"intermediateCRL"`。
4. 外层 `entries` 是一个嵌套 array。
5. 每个条目固定为 `[uid, certFingerprintBytes, revokedAt, reasonCode]`。
6. `signature` 字段本身不进入该数组。

## 9. 匹配与验证规则

接收端在验证某张 User Certificate 是否被 Intermediate CRL 吊销时，应执行以下步骤：

1. 验证 Intermediate CRL 的 Ed25519 签名。
2. 检查 `type` 是否为 `intermediateCRL`。
3. 检查 `issuerSn` 是否与该 User Certificate 的 `issuerSn` 一致。
4. 检查当前时间是否已超过 `nextUpdate`；若超过，则应尽快拉取新 CRL。
5. 按主协议规则重建待验证 User Certificate 的签名数组，并计算其 `certFingerprint`。
6. 在 Intermediate CRL 条目中先按 `uid` 定位，再同时匹配 `certFingerprint`。
7. 当且仅当 `uid` 与 `certFingerprint` 均匹配时，认定该 User Certificate 已被吊销。

说明：

1. 仅 `uid` 匹配而 `certFingerprint` 不匹配时，不应直接视为命中；这通常意味着旧证书已被吊销，但当前证书可能是该 UID 后续重新签发的新证书。
2. 仅 `certFingerprint` 匹配而 `uid` 不匹配时，应视为实现错误或数据损坏。

## 10. 发布与缓存建议

1. Intermediate CRL 可通过 HTTPS、对象存储或静态文件服务发布。
2. Intermediate CA 证书对象中的 `crl` 字段应指向该 Intermediate CRL 的可下载地址。
3. 设备或服务端应缓存最近一次验签通过且 `crlNumber` 最大的 Intermediate CRL。
4. 新下载的 Intermediate CRL 必须先验签，再比较 `crlNumber`，最后替换缓存。
5. 若缓存中的 Intermediate CRL 已过 `nextUpdate` 且无法获取新 CRL，系统可按实现策略降级处理，但应明确记录告警。

## 11. 最小实现建议

如果当前只追求最小可实现版本，至少应做到：

1. 生成和解析 Intermediate CRL JSON。
2. 构造 Intermediate CRL 的 CBOR 签名数组。
3. 使用 Intermediate CA 私钥离线签名，使用 Intermediate CA 公钥验签。
4. 对 User Certificate 计算 `certFingerprint`。
5. 依据 `uid + certFingerprint` 做吊销命中判断。

这样即可在不暴露呼号、不永久拉黑 UID 的前提下，完成 V4 的用户证书级吊销闭环。
